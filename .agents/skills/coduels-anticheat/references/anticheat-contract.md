## Pipeline

1. Frontend records editor and judging actions during an active duel.
2. `actionQueue.ts` adds UUID `event_id`, per-actor `sequence_id`, and an ISO timestamp.
3. Frontend batches up to 200 events and flushes every 5 seconds and on page hide/unload to `POST /actions` with the bearer token.
4. Duely verifies the authenticated user owns the events, ignores actions for finished duels, and persists valid events in PostgreSQL using EF inheritance/discriminators.
5. When a duel ends, pending `AnticheatScore` rows are processed in batches of 20.
6. Duely loads actions ordered by sequence for each `(duel, user, task_key)`, supplies the user's initial duel rating, and calls Analyzer `POST /predict`.
7. Analyzer extracts the ordered feature row, calls `predict_proba`, and returns the positive-class probability as `score`.
8. Duely stores the score and may delete the underlying actions depending on `ShouldCleanupUserActions`.

## Current action types

- `ChooseLanguage`: `language`.
- `WriteCode`: `code_length`, `cursor_line`.
- `DeleteCode`: `code_length`, `cursor_line`.
- `PasteCode`: `code_length`, `cursor_line`, `begin_line`, `end_line`, `chars_count`.
- `CutCode`: same clipboard fields as paste.
- `MoveCursor`: `code_length`, `cursor_line`, `previous_cursor_line`.
- `RunSampleTest`: common fields only.
- `RunCustomTest`: common fields only.
- `SubmitSolution`: common fields only.

Common wire fields:

- `event_id`: UUID string generated by Frontend.
- `sequence_id`: positive integer, increasing per duel/user actor queue.
- `timestamp`: ISO timestamp.
- `duel_id`, `user_id`: positive identifiers.
- `task_key`: exactly one character for persisted/scored actions.
- `type`: exact discriminator string listed above.

## Feature vector

The current ordered `FEATURE_NAMES` tuple is:

1. `typing_speed_avg`
2. `typing_speed_std`
3. `active_intervals_count`
4. `avg_active_interval_duration`
5. `edits_count`
6. `avg_edit_size`
7. `max_edit_size`
8. `edit_size_std`
9. `small_edit_ratio`
10. `paste_count`
11. `avg_paste_size`
12. `max_paste_size`
13. `paste_ratio`
14. `edits_after_paste_ratio`
15. `cursor_jump_count`
16. `typed_chars_ratio`
17. `typed_chars_total`
18. `runs_sample_tests_count`
19. `runs_custom_tests_count`
20. `submit_count`
21. `user_rating`

Feature defaults/configuration:

- typing inactivity threshold: 1 second;
- active interval inactivity threshold: 15 seconds;
- small edit threshold: 5 characters;
- cursor jump threshold: more than 3 lines;
- empty ratios/aggregates become zero.

The extractor sorts by `sequence_id`; training and inference both build DataFrames with `FEATURE_NAMES` order. Renaming, inserting, removing, or reordering a feature invalidates existing models.

## Models and artifacts

`train.py` trains:

- baseline logistic regression -> `artifacts/baseline_logreg.pkl` and metrics JSON;
- production random forest -> `artifacts/prod_random_forest.pkl` and metrics JSON.

The API tries the production model first and baseline second. Expected paths can be overridden with:

- `ANALYZER_PROD_MODEL_PATH`
- `ANALYZER_BASELINE_MODEL_PATH`

Artifacts are ignored by Git and generated in CI before the Analyzer image is built. Local Docker/API startup therefore requires local training first.

## Interpretation

The score estimates how atypical/suspicious the observed behavior is under a small, project-specific dataset. The thesis reports high recall but a false-positive rate around 0.2 and notes the dataset's limited size. Treat the score as a review signal, not a factual accusation.

Evaluate threshold changes as an explicit recall/false-positive tradeoff and keep the raw probability available.

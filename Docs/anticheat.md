# Античит

## Постановка задачи

Разработать метод выявления подозрительного поведения пользователей при участии в дуэлях.

## Ограничения

Следует отметить, что задача не предполагает точного определения факта списывания. Модель оценивает степень отклонения поведения пользователя от типичного процесса написания решения и может использоваться как инструмент для выявления подозрительных случаев, требующих дополнительной проверки.

## Анализ задачи

Пользователь полностью пишет код в редакторе кода на странице браузера, поэтому есть возможность анализировать взаимодействие с редактором кода, а также системой тестирования решений.

Формально, есть последовательность событий $E = \{e_1, e_2, \dots, e_n\}$, где $e_i$ - событие с временной меткой и типом. Требуется построить отображение $f(E) \rightarrow p \in [0, 1]$, где $p$ - степень подозрительности поведения пользователя ($p \approx 0$ - поведение похоже на нормальное, $p \approx 1$ - поведение сильно отклоняется от нормального).

---

### Что считаем подозрительным

Подозрительность = отклонение от типичного процесса разработки решения:
- нет постепенной эволюции кода;
- большие одномоментные вставки без последующей доработки;
- аномально быстрый переход к рабочему решению;
- поведение не похоже на «человеческое» (нет возвратов, правок, дебага).

---

### Важные легитимные сценарии (чтобы не ловить ложные срабатывания)

Нужно учитывать:
- вставка шаблона (особенно в C++) — может быть очень большой;
- вставка своих заготовок (например, алгоритмы);
- копирование/перемещение собственного кода внутри файла;
- дописывание решения после частичной реализации;
- быстрые пользователи (высокий рейтинг).

Вывод: сам факт вставки ≠ подозрительность, важен контекст

---

### Поведенческие паттерны (ключевые сигналы)

Процесс написания:
- код пишется не строго линейно

Пользователь:
- возвращается назад;
- редактирует старые части;
- вставляет куски и дорабатывает их.

Вставки:
- нормальные:
    - шаблон;
    - свои заготовки;
- подозрительные:
    - большой кусок + нет изменений после;
    - большой кусок + быстрый AC.

Ввод:
- typing vs paste (Monaco умеет);
- скорость ввода зависит от пользователя (важен рейтинг).

Скобки:
- обычно:
    - либо автодополнение;
    - либо быстрая пара;
- можно анализировать время между открывающей и закрывающей.

Этап дебага (очень важный сигнал “нормальности”):
- запуск код на тестах;
- движение курсор по коду;
- внесение небольших правок;
- добавление debug-вывода;
- постепенное исправление ошибок.

---

### Сценарии читинга

1. Через ChatGPT
- вставка кода → ловится через:
    - большие paste
    - отсутствие эволюции
    - переписывание руками → сложнее, но может не быть естественных правок
2. Автоматическая генерация (агент печатает):
    - самое сложное
    - можно ловить частично по “нечеловеческой” структуре правок
    - полностью не решается → это нормально

---

### Итоговая идея

Мы отлавливаем не чит, а нарушение естественного процесса разработки решения. И этот процесс характеризуется:
- постепенностью;
- итеративностью;
- наличием дебага;
- неоднородными правками.

Подозрительным считается поведение, при котором отсутствует характерная для разработки решения постепенная эволюция кода: наблюдаются крупные вставки без последующей доработки, аномально быстрое получение результата или нетипичная структура редактирования. При этом учитываются легитимные сценарии, такие как вставка шаблонов или собственных заготовок.

---

#### Поведенческие признаки честного поведения

1. Итеративное написание:
    - код появляется постепенно;
    - пользователь пишет, потом дополняет, потом возвращается и улучшает;
    - нет резких скачков (ничего -> почти всё решение).
2. Возвраты и доработки:
    - пользователь возвращается к уже написанным частям;
    - меняет, переписывает, улучшает;
    - не только дописывает в конец файла.
3. Вставки с адаптацией:
    - может вставить кусок (шаблон/структуру);
    - после этого дорабатывает его под задачу;
    - вставленный код встраивается в решение.
4. Наличие дебага:
    - запускает код на тестах;
    - после запусков делает небольшие правки;
    - может добавлять и убирать debug-вывод.
5. Нелинейная структура работы:
    - работает с разными частями программы;
    - не пишет строго сверху вниз без возвратов;
    - структура кода формируется постепенно.
6. Человеческий ритм редактирования:
    - изменения разного размера;
    - есть мелкие правки и уточнения;
    - нет только больших однотипных действий.

---

#### Поведенческие признаки подозрительного поведения

1. Вставка готового решения:
    - появляется большой кусок кода сразу;
    - до этого почти нет активности;
    - после вставки почти нет изменений.
2. Отсутствие эволюции:
    - код возникает крупными блоками;
    - мало промежуточных состояний;
    - нет постепенного развития решения.
3. Минимум взаимодействия с кодом:
    - мало правок;
    - нет возвратов к написанному;
    - нет доработок существующего кода.
4. Скачкообразный процесс:
    - редкие действия, но очень крупные;
    - нет мелких шагов между ними.
5. Отсутствие дебага:
    - почти нет запусков перед отправкой;
    - нет серии запустил -> поправил -> запустил;
    - решение появляется без явного процесса проверки.
6. Подозрительное поведение после вставки:
    - вставка -> почти сразу запуск или отправка;
    - вставка -> минимальные правки -> финал.
7. Неестественная структура редактирования:
    - нет "живой" работы с кодом;
    - нет локальных изменений;
    - процесс слишком гладкий и прямолинейный.

### Преобразование данных

Последовательность событий $E$ будет преобразована к виду, который будет удобен для работы $t(E) \rightarrow x \in \R^d$. Другими словами, последовательность событий будет преобразована к табличному виду, будет произведено выделение признаков (feature extraction).

#### 1. `typing_speed_avg`

**Что показывает:** среднюю скорость ручного ввода кода пользователем.

**Как считается:**
Берутся только события ручного ввода символов (`typing`), без вставок (`paste`). Временной ряд разбивается на активные интервалы редактирования: если между двумя соседними событиями редактирования прошло больше заданного порога неактивности, начинается новый интервал. Для каждого интервала считается:
$$
speed_i = \frac{\text{число символов, введённых вручную в интервале}}{\text{длительность интервала}}
$$
После этого берётся среднее значение по всем активным интервалам.

**Зачем нужна:**
Позволяет оценить характер ручного набора и отделять его от сценариев, где код в основном появляется не через печать.

---

#### 2. `typing_speed_std`

**Что показывает:** разброс скорости ручного ввода.

**Как считается:**
Для всех активных интервалов редактирования, по которым посчитаны скорости $speed_i$, вычисляется стандартное отклонение.

**Зачем нужна:**
Реальный процесс набора обычно не полностью равномерен. Фича помогает описать ритм ручного ввода.

---

#### 3. `active_intervals_count`

**Что показывает:** на сколько отдельных этапов был разбит процесс написания кода.

**Как считается:**
Временной ряд редактирования разбивается на активные интервалы. Если пауза между двумя соседними событиями редактирования больше заданного порога, например 10–20 секунд, это считается границей между интервалами. Фича равна количеству таких интервалов.

**Зачем нужна:**
Честное решение часто состоит из нескольких этапов: написал часть, остановился, продолжил, доработал.

---

#### 4. `avg_active_interval_duration`

**Что показывает:** среднюю длительность активного этапа редактирования.

**Как считается:**
Для каждого активного интервала считается его длительность:
$$
duration_i = t^{end}_i - t^{start}_i
$$
Далее берётся среднее по всем интервалам.

**Зачем нужна:**
Позволяет описать структуру работы пользователя: длинные сессии набора, короткие рывки или смешанный режим.

---

#### 5. `edits_count`

**Что показывает:** общее количество изменений кода.

**Как считается:**
Считается число всех событий, которые изменяют текст программы: ввод, удаление, замена, вставка.

**Зачем нужна:**
Отличает постепенную разработку решения от сценария, где код появляется небольшим количеством крупных действий.

---

#### 6. `avg_edit_size`

**Что показывает:** средний размер одного изменения кода.

**Как считается:**
Для каждого события редактирования считается абсолютный размер изменения в символах:
$$
edit_size_i = |\Delta_i|
$$
После этого вычисляется среднее:
$$
avg_edit_size = \frac{1}{n}\sum_{i=1}^{n} edit_size_i
$$

**Зачем нужна:**
Позволяет различать процесс с большим количеством мелких правок и процесс, в котором код добавляется более крупными блоками.

---

#### 7. `max_edit_size`

**Что показывает:** размер самого крупного единичного изменения.

**Как считается:**
Берётся максимум по размерам всех изменений:
$$
max_edit_size = \max_i edit_size_i
$$

**Зачем нужна:**
Крупное единичное изменение — важный сигнал, особенно в сочетании с малым количеством последующих правок.

---

#### 8. `edit_size_std`

**Что показывает:** разброс размеров изменений.

**Как считается:**
По всем значениям $edit_size_i$ вычисляется стандартное отклонение.

**Зачем нужна:**
Честный процесс обычно неоднороден: есть мелкие исправления, средние вставки, удаления, локальные доработки.

---

#### 9. `small_edit_ratio`

**Что показывает:** долю маленьких изменений среди всех изменений.

**Как считается:**
Задаётся порог малого изменения, например 1–5 символов. Затем считается:
$$
small_edit_ratio =
\frac{\text{число изменений, размер которых не превышает порог}}{\text{общее число изменений}}
$$

**Зачем нужна:**
Мелкие правки и уточнения характерны для живой работы с кодом.

---

#### 10. `paste_count`

**Что показывает:** сколько раз пользователь вставлял код.

**Как считается:**
Считается количество событий `paste`, которые редактор позволяет отличить от ручного ввода.

**Зачем нужна:**
Фиксирует сам факт использования вставок, что важно в сочетании с другими признаками.

---

#### 11. `avg_paste_size`

**Что показывает:** средний размер вставки.

**Как считается:**
Для каждого события `paste` считается число вставленных символов, после чего берётся среднее.

**Зачем нужна:**
Помогает отделять небольшие вставки от регулярного переноса больших фрагментов кода.

---

#### 12. `max_paste_size`

**Что показывает:** размер самой крупной вставки.

**Как считается:**
Берётся максимум по размерам всех вставок.

**Зачем нужна:**
Одна большая вставка может быть нормальной, но в ряде случаев это важный сигнал, особенно если после неё почти нет доработки.

---

#### 13. `paste_ratio`

**Что показывает:** долю добавленного кода, появившегося через вставку.

**Как считается:**
$$
paste_ratio =
\frac{\text{общее число символов, добавленных через paste}}{\text{общее число символов, добавленных в код}}
$$

Здесь в знаменатель входят все добавленные символы: и введённые вручную, и вставленные.

**Зачем нужна:**
Позволяет оценить вклад вставок в общий процесс написания решения.

---

#### 14. `edits_after_paste_ratio`

**Что показывает:** насколько часто вставленные блоки потом редактировались.

**Как считается:**
Для каждой вставки фиксируется диапазон текста, который был вставлен. Затем проверяется, были ли после этого изменения внутри данного диапазона. После этого считается:
$$
edits_after_paste_ratio =
\frac{\text{число вставок, после которых вставленный блок редактировался}}{\text{общее число вставок}}
$$

**Зачем нужна:**
Это одна из ключевых фич. Вставка шаблона или своей заготовки обычно сопровождается адаптацией, а вставка готового решения — нет.

---

#### 15. `cursor_jump_count`

**Что показывает:** сколько раз пользователь существенно перемещался в другую часть кода.

**Как считается:**
Рассматриваются последовательные позиции курсора в событиях редактирования и навигации. Если расстояние между соседними позициями превышает порог, например несколько строк или заданное число символов, это считается прыжком:
$$
|pos_i - pos_{i-1}| > T
$$
Фича равна количеству таких прыжков.

**Зачем нужна:**
Честный пользователь часто возвращается к уже написанным участкам программы, а не только двигается линейно вперёд.

---

#### 16. `typed_chars_ratio`

**Что показывает:** долю кода, написанного вручную, а не вставленного.

**Как считается:**
$$
typed_chars_ratio =
\frac{\text{число символов, введённых вручную}}{\text{общее число добавленных символов}}
$$

где в знаменатель входят символы, введённые вручную и вставленные через `paste`.

**Зачем нужна:**
Позволяет понять, насколько решение формировалось именно за счёт ручного набора.

---

#### 17. `typed_chars_total`

**Что показывает:** общее количество символов, введённых вручную.

**Как считается:**
Суммируются все символы, добавленные через события ручного ввода, без учёта вставок.

**Зачем нужна:**
Даёт общий масштаб ручной работы пользователя и полезна в сочетании с другими фичами.

---

#### 18. `runs_sample_tests_count`

**Что показывает:** сколько раз пользователь запускал код на тестах из условия.

**Как считается:**
Считается количество запусков, в которых использовались тесты из условия задачи.

**Зачем нужна:**

### Моделирование подозрительности

Модель машинного обучения по набору $x \in \R^d$ моделирует степень подозрительности пользователя $s(x) \rightarrow p \in [0, 1]$. В итоге $f(E) = s(t(E))$.

#### Подход к выбору моделей

Задача моделирования формулируется как задача регрессии: по набору признаков требуется предсказать числовую оценку подозрительности.

При этом из-за отсутствия точной разметки «чит / не чит» задача носит эвристический характер: модель должна выявлять отклонения от типичного поведения, а не строго классифицировать пользователей.

В рамках работы используются два уровня моделей:
- базовая модель (baseline);
- основная (финальная) модель.

---

#### Базовая модель (baseline)

В качестве базовой модели используется логистическая регрессия.

Преимущества:
- простота реализации;
- высокая интерпретируемость;
- малый риск переобучения;
- позволяет быстро проверить корректность признаков.

Логистическая регрессия моделирует зависимость:
$$
p = \sigma(w^T x + b),
$$
где $\sigma$ — сигмоидальная функция.

Использование baseline-модели позволяет:
- убедиться, что выбранные признаки действительно содержат полезную информацию;
- получить начальную оценку качества;
- проанализировать вклад отдельных признаков в итоговый результат.

---

#### Основная модель

В качестве основной модели используется ансамблевый метод — случайный лес (Random Forest).

Причины выбора:
- способность моделировать нелинейные зависимости;
- устойчивость к выбросам и шуму;
- автоматический учёт взаимодействий между признаками;
- отсутствие строгих требований к масштабированию признаков.

Случайный лес строит ансамбль решающих деревьев и усредняет их предсказания:
$$
p = \frac{1}{N} \sum_{i=1}^{N} T_i(x),
$$
где $T_i$ — отдельные деревья решений.

---

#### Обоснование выбора моделей

Выбор логистической регрессии в качестве baseline и случайного леса в качестве основной модели обусловлен следующими соображениями:

- логистическая регрессия позволяет получить интерпретируемую и стабильную базовую оценку;
- случайный лес способен учитывать сложные зависимости между признаками, которые характерны для поведенческих данных;
- обе модели хорошо работают с табличными данными и не требуют сложной предварительной обработки;
- данные имеют умеренную размерность, что делает использование сложных моделей (например, нейронных сетей) избыточным.

---

#### Особенности обучающих данных

Обучение модели осложняется отсутствием точной разметки. В рамках работы возможны следующие подходы:

- использование эвристической разметки (например, выделение явно подозрительных сценариев);
- использование синтетических данных (имитация читинга);
- ручная разметка части данных;
- обучение модели на выявление отклонений от типичного поведения.

Таким образом, модель фактически обучается различать «типичное» и «нетипичное» поведение пользователей.

---

#### Использование модели

После завершения дуэли:
1. из последовательности событий $E$ вычисляется вектор признаков $x = t(E)$;
2. модель вычисляет значение $p = s(x)$;
3. результат сохраняется в базе данных.

Полученное значение не используется для автоматического наказания пользователя, а служит индикатором подозрительности и может быть использовано:
- для анализа поведения пользователей;
- для выявления подозрительных случаев;
- как дополнительный сигнал для модерации.
